# Описание к Курсовой

## Цель

Обучить модель для рекомендации пользователям покупок в интернет-магазине. Модель должна анализировать историю покупок пользователей и находить схожее поведение между разными пользователями. Например, если пользователи покупали товар N и с ним часто покупали товар M, то новому пользователю при покупке товара N будет тоже предложен товар M.

В рамках развития работы реализованы **четыре уровня моделей**:

1. **Baseline** — классическая матричная факторизация ALS.
2. **Two-Stage** — двухступенчатая архитектура (Retrieval + Ranking) на базе ALS и LightGBM.
3. **Sequential (SASRec)** — нейросетевая модель на базе Transformer'ов для учёта последовательностей в поведении пользователей.
4. **Сравнительный анализ** — сопоставление всех моделей по расширенному набору метрик.

Дополнительно внедрены:

- **Time Decay** (временное затухание) для учёта давности событий.
- **Sessionization** (разметка сессий) для выделения осмысленных последовательностей.
- **Централизованное логирование** всего вывода в файл с дублированием в консоль.

## Используемые данные

Дата-сет [Retail Rocket](https://www.kaggle.com/datasets/retailrocket/ecommerce-dataset?resource=download).

Данный дата-сет содержит набор исторических данных о покупках в интернет-магазинах. На основании этих данных можно обучить модель неявным связям в поведении пользователей.

## Примечание

Git не позволяет залить в репозиторий дата-сет. Его можно скачать по ссылке выше. По умолчанию файлы из дата-сета располагаются в `resources`.

## Данные

- События (`events.csv`): `visitorid`, `itemid`, `event` (`view`, `addtocart`, `transaction`), `timestamp`.
- Категории (`item_properties_part1.csv`, `item_properties_part2.csv`): `categoryid`.

## Модель

### 1. Baseline: ALS (Alternating Least Squares)

Классическая матричная факторизация из библиотеки `implicit`. Данная модель часто используется в рекомендательных системах как быстрый и надёжный базовый уровень (Candidate Generation).

### 2. Two-Stage: ALS + LightGBM

Двухступенчатая архитектура:

- **Stage 1 (Retrieval)**: ALS быстро отбирает из десятков тысяч товаров ~500 кандидатов.
- **Stage 2 (Ranking)**: LightGBM переранжирует кандидатов, используя дополнительные признаки (популярность товара, активность пользователя, ALS-скор), и выдаёт финальный Топ-K.

### 3. Sequential: SASRec (Self-Attentive Sequential Recommendation)

Нейросеть на базе механизма внимания (Transformer), реализованная на PyTorch. Анализирует последовательности действий пользователя внутри сессии. В отличие от ALS, SASRec понимает паттерны вида «телефон → чехол → зарядка» и предсказывает следующий товар в сессии.

### 4. Логирование

Весь вывод программы (включая `print()`, предупреждения и ошибки с полным traceback) автоматически сохраняется в файл `recommendation_log_YYYYMMDD_HHMMSS.txt` с дублированием в консоль. Реализовано через перехват встроенной функции `print` и стандартный модуль `logging`.

## Описание реализации

### Базовые веса событий

Для оценки востребованности товара пользователем были выбраны следующие события из `events` и веса:
- `view` — просмотр товара. Вес 1.
- `addtocart` — добавление товара в корзину. Вес 3.
- `transaction` — покупка товара. Вес 5.

Таким образом приоритет в рекомендациях отдаётся товарам, которые были куплены, затем добавленным в корзину, затем просмотренным.

### Time Decay (временное затухание)

Применяется экспоненциальное затухание веса события в зависимости от его давности. Просмотр товара вчера учитывается сильнее, чем просмотр месяц назад.

### Разметка сессий (Sessionization)

История пользователя разбивается на сессии: если между двумя последовательными событиями прошло более 30 минут, считается, что началась новая сессия. Это позволяет Sequential-моделям (SASRec) обучаться на коротких осмысленных последовательностях, а не на всей истории пользователя.

### Фильтрация пользователей

Для создания обучающих и тестовых наборов данных пользователи фильтруются по наличию у них 2-х и более событий.

### Фильтрация категорий

Перед применением `LabelEncoder` к товарам из `item_properties` выполняется фильтрация: оставляются только те товары, которые присутствовали в `events.csv`. Это решает проблему `ValueError: y contains previously unseen labels`, так как `item_encoder` обучен только на товарах из событий.

## Формирование матриц

Матрицы формируются вида «пользователи × товары», где ячейка = итоговый вес взаимодействия (с учётом Time Decay).

## Разделение на тестовые данные

Для тестовых данных берётся последнее событие пользователя.

## Рекомендации

Для заданного пользователя система:
- берёт его эмбеддинг,
- умножает на эмбеддинги всех товаров,
- сортирует по убыванию,
- исключает уже просмотренные,
- возвращает топ самых релевантных.

В Two-Stage подходе после ALS-отбора кандидатов финальное ранжирование выполняется градиентным бустингом LightGBM на кросс-признаках.

В SASRec рекомендации формируются на основе последнего скрытого состояния Transformer'а, полученного из последовательности сессии пользователя.

## Метрики качества

### `recall_at_k`

Функция оценивает качество рекомендаций на тестовых данных. Какая доля реальных покупок (из теста) попала в топ-K рекомендаций.

### `ndcg_at_k` (Normalized Discounted Cumulative Gain)

Учитывает **порядок** рекомендаций: релевантный товар на 1-м месте оценивается выше, чем на 10-м. Позволяет штрафовать модель за «закапывание» нужных товаров.

### `catalog_coverage`

Показывает, какую долю всего каталога модель когда-либо рекомендует. Помогает выявить проблему «залипания» на популярных товарах.

### `category_diversity`

Доля уникальных категорий в топ-K рекомендациях. Борется с «мыльным пузырём», когда модель рекомендует 10 однотипных товаров.

## Описание функций

### Метрики

- `recall_at_k` — оценивает долю реальных покупок из теста, попавших в топ-K рекомендаций.
- `ndcg_at_k` — вычисляет нормализованный кумулятивный выигрыш с учётом позиции релевантного товара.
- `catalog_coverage` — оценивает долю каталога, которую модель когда-либо рекомендовала.
- `category_diversity` — измеряет разнообразие категорий в топ-K рекомендаций.
- `evaluate_model_advanced` — расширенная оценка модели: возвращает Recall@K, NDCG@K, Coverage и Diversity.

### Feature Engineering

- `apply_time_decay_and_sessions` — применяет временное затухание весов событий и размечает историю пользователей на сессии с порогом 30 минут.

### Архитектуры моделей

- `TwoStageRecSys` — класс, реализующий двухступенчатую архитектуру: ALS для Retrieval и LightGBM для Ranking.
- `SASRec` — нейросетевой модуль (PyTorch) с механизмом внимания для предсказания следующего товара в сессии.
- `SessionDataset` — PyTorch Dataset для подготовки сессий к обучению SASRec.
- `train_sasrec` — функция обучения SASRec.
- `evaluate_sasrec` — функция оценки SASRec на тестовых сессиях.

### Вспомогательные

- `load_item_categories` — получает данные о связках товаров и категорий. Отбрасывает все ненужные поля из `item_properties`.
- `visualize_item_embeddings` — строит 2D-график, где каждый товар — точка, раскрашенная по категории. Отображается топ-10 категорий.
- `setup_logger` / `print_with_logging` — настройка логгера и перехват `print()` для сохранения всего вывода в файл.

## Вывод

1. **Baseline ALS** определил семантическое сходство товаров из данных о просмотрах и покупках. Однако, как и в прошлой версии, чётких кластеров по категориям не наблюдается — пользователи покупают товары из разных категорий, а поведение в датасете довольно однотипное.

2. **Time Decay и Sessionization** позволили модели фокусироваться на «свежих» сессиях. Средний вес событий после затухания снизился, что подтверждает корректную работу экспоненциального коэффициента. Выделено значительное количество сессий, пригодных для обучения Sequential-моделей.

3. **Продвинутые метрики** показали ограничения Recall@K:

   - `NDCG@K` выявил, что ALS часто размещает релевантные товары не на первых позициях.
   - `Coverage` показал, что ALS рекомендует лишь небольшую долю каталога (в основном популярные товары).
   - `Diversity` подтвердил проблему «мыльного пузыря» — в топ-10 часто попадают товары одной категории.

4. **Two-Stage ALS + LightGBM** улучшил ранжирование кандидатов за счёт нелинейных признаков (популярность, активность пользователя). Это стандартная индустриальная архитектура, позволяющая гибко добавлять новые признаки без переобучения ALS.

5. **SASRec** на базе Transformer'ов показал прирост метрик по сравнению с Baseline ALS, особенно по `NDCG@K`. Модель научилась учитывать порядок действий в сессии и предсказывать следующий товар на основе контекста, а не только статистической схожести.

6. **Логирование** обеспечило надёжную диагностику: все ошибки (включая `ValueError: unseen labels` из-за расхождения наборов товаров в `events` и `item_properties`) теперь фиксируются в файле с полным traceback, что значительно упрощает отладку.

7. **Сложности и дальнейшие направления**:
   - Для улучшения кластеризации требуется дополнительная классификация категорий (например, с помощью LLM для построения иерархической таксономии).
   - Целесообразно внедрение Graph Neural Networks (LightGCN) для учёта связей между пользователями и товарами.
